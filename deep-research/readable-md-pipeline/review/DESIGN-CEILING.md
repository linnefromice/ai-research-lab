# 見た目をコードでどこまで改善できるか — デザイン自動化の天井

生成物 (HTML/スライド) の「見た目 human friendly」を、人間の目や taste に頼らず
**コードでどこまで改善できるか**を検討し、到達点と壁を地図化する。これは実装前の
設計ドキュメントで、`review/` バンドルをどこまで育てるかの指針にする。

## 結論

- **「正しさ」の軸 (内部一貫性・アクセシビリティ・原則準拠) はコードで実質 100% 保証できる。**
- **「適切さ」の軸 (意味に対する見た目の妥当性・ブランド・意図) は人間が seed を選ぶ。**
- ノンデザイナーズ・デザインブックの C.R.A.P. は **ルール**であり、ルールは全部コード化できる。
  「目が要る」と思われがちな Alignment / Proximity も、**レンダリングして座標を測れば計算可能**。
- 我々のように「1 テーマ × 構造化 md × 既知コンポーネント」の制約系では設計空間が小さく、
  天井はかなり高い。

## 中心となる区別: 正しさ vs 適切さ

| 軸 | 例 | コードで | 担当 |
|----|----|---------|------|
| **正しさ (correctness)** | コントラスト AA・type scale の段差・余白の規律・整列・溢れなし | **ほぼ 100% 自動** | machine |
| **適切さ (appropriateness)** | この内容/読者にこのトーンは合うか・何を一番強調すべきか | 不可 | human (seed) + LLM |

コードは「何かが目立つ」は検証できるが「**目立たせるべき正しい要素か**」は内容理解が要る。
ここが分水嶺。

## コード化の梯子 (Lv0–4)

| Lv | 測れる/直せること | 必要なもの | 自動修正 | 決定論 | 現状 |
|----|------------------|-----------|---------|--------|------|
| **0 静的トークン監査** | コントラスト比・行長・type scale・パレット | なし | 提案のみ | ○ | 構築済 (`design-audit.sh`) |
| **1 静的デザインシステム強制/生成** | modular scale 準拠・8px グリッド・パレット由来・vertical rhythm | なし | **可 (asset 書換/生成)** | ○ | 構築済 (`gen-theme.sh`) |
| **2 レンダリング幾何** | Alignment/Proximity 数値化・overflow・衝突・print page-fit・レスポンシブ | headless chromium | 一部 | ○ | 未 |
| **3 ピクセル/知覚** | 視覚バランス(重心)・clutter(乱雑度)・実パレット抽出・色調和 | render + 画像処理 | 一部 | ○ | 未 |
| **4 学習/taste** | 美的スコア・ブランド適合・C.R.A.P. 定性 | 美的モデル / LLM-vision | 不可 | × | skill 雛形のみ |

### Lv0 静的トークン監査 (構築済)

CSS トークンを解析し、レンダリングなしで測れるもの。`design-audit.sh` が担う。
コントラスト比 (hex→WCAG 相対輝度)、行長 (measure÷font-size)、type scale、パレット数。
**測るだけで直さない**。

### Lv1 静的デザインシステム強制/生成 (新依存なしで一番効く)

ルールに対する適合を強制し、さらに **seed から全体を生成**する。

- **強制 (lint)**: ハードコード色の禁止 (全色は token 由来)、余白が 8px グリッドに乗っているか、
  type scale が modular scale (例 1.25 等比) の段に乗っているか、line-height が baseline の倍数か。
- **生成 (generate)**: 少数の seed から決定論的に全体を導出する。これが「どこまで行けるか」の核心。

```text
seed:  base=16px  ratio=1.25  space=8px  hue=245
  ↓ 決定論的に展開
type scale : 16 / 20 / 25 / 31 / 39 px        (base × ratio^n)
spacing    : 8 / 16 / 24 / 32 / 48 px         (space × n)
palette    : OKLCH で hue を起点に明度/彩度を回し、
             全 fg/bg ペアの WCAG コントラストを計算して AA 以上を保証
  ↓
report-theme-head.html を生成
```

Radix Colors / Material You が実際にこの方式。**一貫性とアクセシビリティをコードで保証**できる。

### Lv2 レンダリング幾何 (A/P を「目視」から「計算」へ)

headless ブラウザでレンダリングし、要素の boundingBox を測る。決定論だが chromium が要る。

- **Alignment**: 全ブロックの左端 X 座標を集めてクラスタリング → 使う「揃え線」が少ないほど高得点。
- **Proximity**: 要素間の縦ギャップを測り「グループ内 < グループ間」かを検査。
- **Overflow**: コンテンツが枠/スライドを超えていないか (Marp の溢れは実バグ源)。
- **衝突**: box の重なり検出。
- **print page-fit**: PDF 化してページ数・カード途中改ページ・見出しの孤立 (orphan) を検出。
- **レスポンシブ**: 複数幅でレンダリングして崩れ検出。

> ここが重要な飛躍。「人間の目が要る」とされがちな A/P は、座標を測れば**決定論で数値化できる**。
> `design-review` スキルの「目視」記述は Lv2 未導入時の fallback であり、本質的限界ではない。

### Lv3 ピクセル/知覚 (画像処理・重め)

スクリーンショットを撮り、画像から知覚的メトリクスを計算する。

- 視覚的バランス (ink の重心が中心か)、clutter (feature congestion / subband entropy など研究指標)、
  実際の描画パレット抽出、色調和 (補色/類似色)、ピクセル単位のコントラスト (画像/グラデ上の文字も)。
- 決定論だが効果は逓減しやすい。Lv2 まででかなりカバーできる。

### Lv4 学習/taste (コードの外)

美的予測モデルや LLM-vision による定性評価。seed (トーン/ブランド) の選択もここ。
**ここだけは人間と学習モデルの領分**。`design-review` スキルが受け持つ。

## C.R.A.P. がどのレベルで計算可能になるか

| 原則 | 計算可能になる Lv | 測り方 |
|------|------------------|--------|
| **Contrast** | Lv0 (トークン) → Lv3 (ピクセル) | WCAG 比。実描画上はピクセルで |
| **Repetition** | Lv0 (インライン style 検出) → Lv2 (computed-style クラスタ) | 同役割要素の計算済みスタイル一致 |
| **Alignment** | **Lv2** | 左端 X 座標のクラスタ数 |
| **Proximity** | **Lv2** | ギャップ比 (群内 < 群間) |
| **Typography** | Lv0 / Lv1 | 行長・行間・scale 段差 |

つまり **C.R.A.P. はすべて Lv2 までで計算可能**。残る Lv4 は「正しさ」でなく「適切さ」。

## 本当の壁 (コードが超えられない 4 点)

1. **意味に対する見た目の適切さ**: 全指標満点でも、訃報に陽気なテーマは「正しいが不適切」。
2. **正しい対象の強調**: 焦点が「在る」ことは検証できるが「正しい焦点か」は内容理解が要る。
3. **ブランド/文化的含意**: 色の意味・書体の人格・文化差。
4. **ルールを破る創造**: 指標は既知ルールに対してしか測れない。新規な構図は評価外。

ただしこれらは **seed (意図) の選択**に集約される。seed さえ人間が与えれば、そこからの展開・
検証・一貫性維持はコードが全部やれる。

## なぜ我々の系は天井が高いか

- **設計空間が小さい**: 1 テーマ・既知コンポーネント (`.card`/`.box`/`.vflow`...)・構造化 md。
- **ルール支配的**: 自由な Web デザインでなく「レポート/スライド」の型が決まっている。
- **非デザイナーが必要とするのは「壊れない・一貫している」**で、それはまさにコード化可能な部分。
  デザインブックの主旨 (才能でなくルール) と一致する。

→ 「consistently good, never broken」までは**コードで到達できる**。その先の「素晴らしい/らしい」が
human + 学習の領分。

## 推奨ロードマップ

| 順 | やること | レベル | 依存 | 狙い |
|----|---------|--------|------|------|
| 1 | **gen-theme + style lint** | Lv1 | なし | ✅ gen-theme.sh 構築済 (seed→theme 生成・AA 保証)。style lint は余白 |
| 2 | **geometry-audit** | Lv2 | chromium | A/P を数値化・overflow・print page-fit。C.R.A.P. 完全自動化 |
| 3 | (任意) pixel-metrics | Lv3 | chromium + ImageMagick | バランス/clutter。逓減気味なので必要時 |
| 4 | design-review を seed/taste 層に純化 | Lv4 | LLM-vision | 「適切さ」だけを人間/LLM に残す |

現環境は chromium 未導入なので、**まず 1 (新依存なし) が現実的**。chromium を入れたら 2 で
C.R.A.P. が完全に計算可能になる。

## パイプラインへの組み込み

- `design-audit.sh` (Lv0) は既存。Lv1 の `gen-theme.sh` は **theme asset を生成**する側 (md-to-html の
  単一ソースを供給)。Lv2 の `geometry-audit` は生成物 HTML/slide を検査する側。
- CI: `gen-theme (Lv1) → render (Stage2/3) → design-audit (Lv0) + geometry-audit (Lv2)` を
  gate にすれば、見た目の「正しさ」は push ごとに自動保証される。
- 「適切さ」は人間が seed を選び、節目で `design-review` (Lv4) を回す。
